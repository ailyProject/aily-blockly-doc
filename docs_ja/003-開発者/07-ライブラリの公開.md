# ライブラリの公開

「ライブラリを公開」機能を使用すると、現在のプロジェクトにあるローカルBlocklyライブラリをaily Blocklyの公開ライブラリリポジトリへ提出し、他のユーザーがインストールできるようになります。

aily BlocklyはあなたのGitHubアカウントを使用してPull Request（PR）を作成または更新します。提出成功はシステムがリクエストを受け付けたことを示すだけで、審査通過やマージ完了を意味するものではありません。

## 公開前の準備

次の項目を確認してください：

- aily Blocklyにログインしている；
- 使用可能なGitHubアカウントを持っている；
- ローカルライブラリを現在のプロジェクトへインポートしている；
- 実機でのテストに合格している；
- GitHubへ接続できるネットワーク環境がある；
- ライブラリにパスワード、トークン、非公開アドレスなどの機密情報が含まれていない。

ローカルライブラリをまだインポートしていない場合：

1. ライブラリのテストに使用するプロジェクトを開きます；
2. 「ライブラリ管理」を開きます；
3. 「ローカルライブラリをインポート」をクリックします；
4. ライブラリのルートディレクトリを選択します；
5. インポートが完了するまで待ち、左側のBlocklyツールボックスにライブラリが表示されることを確認します。

## ライブラリファイルの確認

ライブラリのルートディレクトリには、少なくとも次のファイルが必要です：

```
your-library/
├── package.json
├── block.json
├── toolbox.json
└── generator.js
```

次のファイルとディレクトリも用意することを推奨します：

```
your-library/
├── readme.md       // ユーザー向け説明書
├── readme_ai.md    // AI向け説明書、任意
├── src/            // Arduino/C/C++ソースコード、任意
├── i18n/           // 多言語ファイル、任意
└── pinmaps/        // ピンマッピングファイル、任意
```

提出時、aily Blocklyは次の内容を自動的に検証します：

- `package.json`、`block.json`、`toolbox.json`が有効なJSONである；
- `package.json`に文字列型の`name`と`version`があり、パッケージ名がインポート済みライブラリ名と一致している；
- `block.json`の最上位が空でない配列で、各ブロックの`type`が一意である；
- `toolbox.json`のカテゴリとブロック設定が有効である；
- `generator.js`が存在し、JavaScriptの構文エラーがない。

空でない`src/`ディレクトリがある場合、提出時に自動で`src.7z`へ圧縮されます。ライブラリのルートに空でない`src.7z`がすでにある場合は、そのファイルが直接使用されます。

完全なライブラリ構成と開発要件については、「ライブラリ仕様」と「ライブラリ適応」を参照してください。

## 実機でのテスト

公開フォームでは、ライブラリが実機でテスト済みであることの確認が必須です。提出前に少なくとも次の項目を検証してください：

- ブロックが正しく表示、接続され、コードを生成できる；
- 生成コードを対象ボード向けにコンパイルできる；
- ファームウェアを書き込める；
- 主要機能が実機で正しく動作する；
- `package.json`に記載した対応ボードが実際のテスト結果と一致している。

## 公開フォームを開く

1. 左側のBlocklyツールボックスで、インポート済みローカルライブラリを探します；
2. ライブラリのカテゴリ名を右クリックします；
3. 「ライブラリを公開」を選択します。

「ライブラリを公開」はローカルライブラリの右クリックメニューにのみ表示されます。項目が表示されない場合は、「ライブラリ管理 > ローカルライブラリをインポート」から再度インポートしてください。

ログインしていない場合は、「ユーザーセンター」を開いてログインするよう案内されます。ログイン済みでもGitHubのPR権限がない場合は、「認証する」をクリックし、ブラウザで認証を完了してaily Blocklyへ戻ります。

現在のフローでは、PRの作成または更新にGitHubの`repo`権限が必要です。aily Blocklyを開いたまま、可能であれば5分以内に認証を完了してください。認証に成功すると提出が自動的に続行されます。

## 公開情報の入力

| 項目 | 必須 | 入力要件 |
| --- | --- | --- |
| パッケージ名 | はい | `@aily-project/lib-`で始めます。それ以降には小文字、数字、ハイフンのみを使用でき、先頭と末尾は文字または数字にする必要があります。例：`@aily-project/lib-dht11`。 |
| バージョン | はい | `1.0.0`のような数値の`メジャー.マイナー.パッチ`形式を使用します。`v`、日付、プレリリース接尾辞は追加しないでください。 |
| 表示名 | はい | ライブラリ管理とツールボックスでユーザーに表示される名前です。例：「DHT11温湿度センサー」。 |
| ライブラリの説明 | いいえ | 対応ハードウェア、主な機能、利用場面を簡潔に説明します。 |
| PRの説明 | いいえ | 今回の機能、公開する理由、テスト内容を記載します。上限は2000文字です。詳しい説明は審査に役立ちます。 |
| 作者 | いいえ | 個人、チーム、または元ライブラリの作者名を入力します。第三者のオープンソースコードを使用する場合は、`readme.md`に出典とライセンス情報を残してください。 |
| キーワード | いいえ | `sensor, temperature, humidity`のように半角カンマで区切ります。 |

PRの説明には、次のテンプレートを使用できます：

```
主な機能：
- （今回のリリースに含まれる機能）

対応ハードウェアと利用場面：
- （対応ボード、センサー、モジュール）

公開する理由：
- （このライブラリが必要な理由）

テスト内容：
- aily Blocklyのバージョン：
- ボードとコアのバージョン：
- 検証済み機能：
- テスト結果：
```

## ローカル情報を同期するか選択する

公開フォームには、「パッケージ名、バージョン、表示名、説明、作者、キーワードをローカルライブラリの package.json に保存する」の項目があります。初期状態では選択されていません。

- 選択しない：フォームの内容は今回の提出にのみ使用され、ローカルの`package.json`は変更されません；
- 選択する：提出成功後、フォームの内容がローカルライブラリへ保存されます；
- パッケージ名を変更した場合、ローカルライブラリのディレクトリ名と現在のプロジェクトの依存関係も更新されます。

フォームが正式なリリース情報になっていることを確認してから選択することを推奨します。これにより、ローカル情報と提出情報の不一致を防げます。

プロジェクトの再読み込みを求められた場合は、「再読み込み」を選択してください。再読み込みしない場合、ツールボックスと依存関係に古い情報が表示され続けることがあります。

## ライブラリを提出する

1. パッケージ名、バージョン、PRの説明を確認します；
2. 「このライブラリが実機でテスト済みであることを確認します」を選択します；
3. ローカルの`package.json`へ同期するか決めます；
4. 「ライブラリを公開」をクリックします；
5. 提出受付のメッセージが表示されるまで待ちます。公開ボタンを繰り返しクリックしないでください。

提出内容には、パッケージ設定、ブロック定義、コードジェネレーター、説明書、多言語ファイル、ピンマッピングファイル、存在する場合はソースアーカイブが含まれます。

## 提出状況を確認する

提出が受理されると、システムがPRを自動的に作成または更新します。成功メッセージの「ユーザーセンター」をクリックするか、後でユーザーセンターのライブラリ提出履歴を開いて状況を確認できます。

通常、その後に自動チェックと管理者による審査が行われます。GitHubの通知とアカウントのメールを確認してください：

- 承認：ライブラリが公開[aily-blockly-libraries](https://github.com/ailyProject/aily-blockly-libraries)リポジトリへマージされます；
- 変更依頼：PRのレビューコメントに従ってローカルライブラリを修正し、同じパッケージ名でもう一度公開します；
- 却下：理由を確認し、問題を修正して再提出します。

aily Blocklyでは、通常48時間以内に管理者が審査すると案内されています。実際の時間は、提出数、休日、修正回数によって変わる場合があります。

## 提出済みライブラリを更新する

同じパッケージ名をもう一度公開すると、すでに提出済みであることが表示されます：

- 内容に変更がない：続行しても重複したPRは作成されません；
- 内容に変更がある：既存の提出履歴が更新され、バックエンドがPRを再作成または更新します；
- 他のユーザーが名前を使用している：そのライブラリを上書きできません。未使用のパッケージ名へ変更してください。

新しいリリースを公開する前にバージョンを更新し、新機能、修正、対応範囲、テスト環境をPRの説明に記載してください。

## よくある問題

### 右クリックメニューに「ライブラリを公開」がない

この項目はローカルライブラリにのみ表示されます。「ライブラリ管理 > ローカルライブラリをインポート」から再度インポートし、ツールボックスに読み込まれることを確認してください。

### ライブラリ情報を読み取れない、またはファイルが無効と表示される

主な原因：

- `package.json`、`block.json`、`toolbox.json`、`generator.js`のいずれかがない；
- JSONファイルにコメント、末尾の余分なカンマ、対応しない括弧がある；
- `package.json`の`name`がインポート済みライブラリ名と一致しない；
- `block.json`が空、ブロックに`type`がない、または複数のブロックで同じ`type`を使用している；
- `toolbox.json`のカテゴリに`name`または`contents`がない；
- `generator.js`が空、または構文エラーがある。

問題を修正し、プロジェクトを再読み込みしてから再度公開してください。

### パッケージ名がすでに存在する、または使用できない

以前に自分が提出したライブラリであれば、更新を確認して続行できます。他のユーザーのライブラリであれば、パッケージ名を変更してください。フォルダー名だけでなく、`package.json`の`name`も変更する必要があります。公開フォームで名前を変更し、ローカル情報の同期を有効にすると、提出成功後にローカルライブラリとプロジェクトの依存関係が更新されます。

### GitHub認証後に提出が続行されない

ブラウザで認証が完了し、aily Blocklyへ戻ったことを確認してください。5分以上経過した、認証をキャンセルした、GitHubの認証情報が無効、または`repo`権限が不足している場合は、もう一度「ライブラリを公開」を開いて再認証してください。

### `src.7z`の作成に失敗する

`src/`が読み取り可能な空でないディレクトリで、他のプログラムがファイルをロックしていないことを確認してください。手動で作成した`src.7z`を使用する場合は、空でないことと、アーカイブの最上位ディレクトリが`src/`であることを確認してください。

### 情報を同期した後もツールボックスに古い名前が表示される

確認ダイアログで「再読み込み」を選択してください。「後で」を選択した場合は、プロジェクトを保存して開き直すと反映されます。

## 提出前チェックリスト

- [ ] 「ローカルライブラリをインポート」からライブラリを追加した；
- [ ] 4つの必須ファイルが存在し、正しく読み込まれる；
- [ ] パッケージ名が`@aily-project/lib-xxx`規則に従い、他のユーザーに使用されていない；
- [ ] バージョンが`x.y.z`形式で、今回の変更に合わせて更新されている；
- [ ] `readme.md`に使い方、配線、依存関係、第三者ライセンスを記載した；
- [ ] 対象ボードでコンパイル、書き込み、実機動作に成功した；
- [ ] アカウント情報、秘密鍵、その他の機密情報が含まれていない；
- [ ] PRの説明に主な機能、利用場面、テスト環境を記載した；
- [ ] フォームの内容をローカルの`package.json`へ同期するか決めた。

その他の開発・提出要件については、[aily-blockly-librariesのライブラリ仕様](https://github.com/ailyProject/aily-blockly-libraries/blob/main/%E5%BA%93%E8%A7%84%E8%8C%83.md)と[コントリビューションガイド](https://github.com/ailyProject/aily-blockly-libraries/blob/main/CONTRIBUTING.md)を参照してください。
